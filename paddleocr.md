# Документация по использованию PaddleOCR в проекте WorkBangers CRM

## 1. Что такое PaddleOCR?
**PaddleOCR** — это мощная библиотека с открытым исходным кодом от компании Baidu, построенная на базе фреймворка глубокого обучения PaddlePaddle. Она специализируется на оптическом распознавании символов (OCR) и извлечении текста из изображений.

### Что может PaddleOCR:
* **Мультиязычность:** Поддерживает более 80 языков, включая русский (ru) и английский (en), а также их смешанное использование.
* **Распознавание под углом:** Умеет распознавать текст, даже если фотография была сделана криво, под углом или вверх ногами (за это отвечает параметр `use_angle_cls=True`).
* **Точность:** Отлично справляется со сложными фонами, тенями, потертостями на чеках и мелким шрифтом.
* **Структура:** Возвращает не просто текст, а точные координаты каждого слова (bounding boxes) и степень уверенности (confidence score).

### Чего НЕ может PaddleOCR (в базовом виде):
* **Осмысление текста:** Библиотека просто выдает «сырой» текст. Она не понимает, где ИНН, где название магазина, а где сумма налогов.
* **Структуризация (JSON):** Она не может самостоятельно собрать из текста готовую структуру данных (например, разбить чек на позиции). Для этого нужен NLP-инструмент (в нашем случае мы используем OpenAI GPT-4o-mini).
* **Плохой почерк:** PaddleOCR заточен под печатный текст. Рукописный текст он распознает гораздо хуже.
* **Низкое разрешение:** Если картинка слишком размытая и пиксельная, качество распознавания резко падает.

## 2. Как OCR реализован в нашем проекте?

Архитектура системы обработки чеков разделена на несколько изолированных этапов, чтобы интерфейс пользователя не зависал (асинхронность), а сам сервер не падал от перегрузки памяти.

### Этап 1: Загрузка чека (Frontend -> API)
1. Пользователь выбирает файлы чеков в интерфейсе браузера.
2. Фронтенд (`receipts-ocr.js`) мгновенно отображает "мутные" карточки со статусом *«В очереди на распознавание»*, используя локальный `URL.createObjectURL()`.
3. Файлы по одному улетают на `POST /api/v1/receipts/queue`.
4. Laravel API сохраняет картинку на диск сервера в папку `storage/app/receipts_queue` и создает запись в таблице базы данных `receipts_queue` со статусом `pending`.

### Этап 2: Асинхронная обработка (Laravel Queue)
Чтобы не заставлять пользователя ждать, тяжелая работа делегируется фоновому воркеру (`ProcessReceiptOcr`).
1. Воркер берет задачу из базы, меняет статус на `processing` (*«Распознавание...»*).
2. Картинка чека отправляется HTTP-запросом в наш внутренний микросервис на Python (PaddleOCR).

### Этап 3: Распознавание (Docker контейнер `crm_ocr`)
OCR вынесен в отдельный Docker-контейнер, чтобы тяжелые библиотеки Python не засоряли основной веб-сервер.
1. Flask-сервер принимает картинку на внутренний порт `8868`.
2. **Изоляция памяти:** При получении запроса сервер инициализирует новый экземпляр `PaddleOCR`. Это сделано специально (мы решили проблему `could not execute a primitive`), чтобы движок MKLDNN не падал при последовательной передаче картинок разных размеров. Экземпляр уничтожается после ответа.
3. PaddleOCR возвращает сырой текст чека.

### Этап 4: Структуризация через ИИ (OpenAI GPT)
1. Laravel берет сырой текст от PaddleOCR и отправляет его в API OpenAI (`gpt-4o-mini`).
2. GPT получает строгий системный промпт: вычленить из "каши" слов название магазина, налоги, итоговую сумму, дату, метод оплаты и конкретные товары.
3. Возвращается четкий JSON, который мы нормализуем (`normalizeFields`) и сохраняем в таблицу `receipts_queue` в колонку `parsed_data`, меняя статус на `ready` (*«Распознано»*).

### Этап 5: Финализация (Frontend)
1. Фронтенд каждые 3 секунды тихо опрашивает API (`pollQueue`).
2. Заметив, что чек перешел в статус `ready`, карточка становится четкой, загорается кнопка сохранения, и данные подставляются в модальное окно.

## 3. Рекомендации по эксплуатации и улучшению

1. **Очистка временных файлов:**
   В папке `storage/app/receipts_queue` будут копиться исходники фотографий чеков, пока пользователь не нажмет кнопку «Сохранить» (после чего чек улетает на Google Drive, а локальный файл удаляется). Если пользователь просто закроет страницу и забудет про чеки, файлы останутся.
   * **Рекомендация:** Настроить Laravel Cron Job (Task Scheduler), который раз в сутки будет удалять записи из таблицы `receipts_queue` и файлы с диска, если они старше 48 часов.

2. **Ограничение ресурсов для Docker:**
   PaddleOCR при инициализации может потреблять до 1-1.5 ГБ оперативной памяти (RAM).
   * **Рекомендация:** В `docker-compose.yml` желательно установить лимиты по памяти для контейнера `ocr`, чтобы он случайно не "съел" всю память сервера (например, `mem_limit: 2g`).

3. **Оптимизация OCR:**
   Мы создаем экземпляр PaddleOCR на каждый запрос из-за багов с очисткой памяти в MKLDNN.
   * **Рекомендация:** Если в будущем поток чеков станет огромным (сотни чеков в час), имеет смысл перевести Python OCR сервер на библиотеку FastAPI + Celery, либо использовать Gunicorn с флагом `--max-requests 1`, что позволит держать модель в памяти и безопасно перезапускать worker'ов без утечек.

4. **Мультиязычность в OCR:**
   Сейчас в `server.py` жестко задано `lang='en'`. Поскольку чеки в основном канадские, это работает отлично.
   * **Рекомендация:** Если появятся чеки на французском (Квебек) или других языках, параметр нужно будет поменять на смешанный (например, `lang='latin'` или `en+fr`), но это может слегка увеличить время распознавания.

5. **Предобработка картинок:**
   PaddleOCR работает быстрее на черно-белых картинках с высоким контрастом.
   * **Рекомендация:** Перед отправкой изображения в OCR можно добавить в Python скрипт шаг по увеличению контрастности и конвертации в градации серого через `cv2` (OpenCV). Это может ускорить процесс на 10-15% и повысить точность считывания блеклых чеков.
